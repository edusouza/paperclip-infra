## 1. Contexto inicial

**Objetivo:** Rodar o [Paperclip](https://paperclip.ing) (control plane para empresas com agentes de IA)
numa VM no GCP, acessível via domínio próprio no CloudFlare, com toda a infra em código (OpenTofu).

**Decisões tomadas:**

| Decisão | Escolha | Motivo |
|---|---|---|
| Compute | GCE (Compute Engine) | Precisaremos instalar outras coisas na VM além do Paperclip |
| Database | PostgreSQL local na VM | Sem Cloud SQL para simplificar |
| Exposição | Cloudflare Tunnel | Sem Nginx/Certbot — TLS gerenciado pelo CloudFlare |
| Region | us-central1-a | Preferência do usuário |
| Machine | e2-standard-2 (2 vCPU, 8 GB) | Margem confortável acima do mínimo (1 vCPU, 2 GB) |
| Domínio | paperclip.eduardosouza.dev | (depois generalizado para example.com no repo público) |

## 5. Erros encontrados e correções

### 5.1 Provider CloudFlare v4 — recursos renomeados

**Erro:**
```
Warning: Deprecated Resource — `cloudflare_tunnel` is now deprecated
Error: Unsupported argument — http_host_header_rewrite is not expected here
```

**Causa:** Provider CloudFlare v4 renomeou os recursos.

**Correção em `cloudflare.tf`:**
```hcl
# ERRADO (deprecated)
resource "cloudflare_tunnel" "paperclip" { ... }
resource "cloudflare_tunnel_config" "paperclip" { ... }

# CORRETO
resource "cloudflare_zero_trust_tunnel_cloudflared" "paperclip" { ... }
resource "cloudflare_zero_trust_tunnel_cloudflared_config" "paperclip" { ... }
```

Também removido o argumento inválido `http_host_header_rewrite` do bloco `origin_request`.

### 5.2 Referência antiga em outputs.tf

**Erro após renomear recursos:**
```
Error: Reference to undeclared resource
  on outputs.tf line 18: cloudflare_tunnel.paperclip.id
```

**Causa:** `outputs.tf` ainda referenciava o nome antigo após a renomeação.

**Lição:** ao renomear recursos Terraform, buscar o nome antigo em TODOS os arquivos antes de aplicar.

**Correção:** atualizar `outputs.tf` e `gce.tf` para usar `cloudflare_zero_trust_tunnel_cloudflared.paperclip.*`.

### 5.3 APIs do GCP não estavam ativas

**Erro:** `tofu apply` falhou com erro de API não habilitada.

**Correção:** ativar manualmente antes do primeiro apply:
```bash
gcloud services enable compute.googleapis.com --project YOUR_GCP_PROJECT_ID
gcloud services enable iam.googleapis.com --project YOUR_GCP_PROJECT_ID
```

**Lição:** documentar isso como pré-requisito (adicionado ao README).

### 5.4 Chave SSH — usuário não tinha id_ed25519

O usuário tinha `~/.ssh/github` (não o nome padrão). Solução: usar `-i ~/.ssh/github` no ssh,
ou adicionar ao `~/.ssh/config`:
```
Host paperclip
    HostName <IP>
    User ubuntu
    IdentityFile ~/.ssh/github
```

### 5.5 `npx paperclipai onboard --yes` ignora variáveis de servidor

**Erro após startup:**
```
authenticated public exposure requires auth.baseUrlMode=explicit
```

**Causa:** o flag `--yes` força modo `local_trusted/private` e **ignora silenciosamente**
todas as variáveis de deployment (`PAPERCLIP_DEPLOYMENT_MODE`, `PAPERCLIP_DEPLOYMENT_EXPOSURE`,
`PAPERCLIP_AUTH_PUBLIC_BASE_URL`). O serviço systemd ficou em loop de restart (restart counter = 39).

**Nunca usar `--yes` para deploy em servidor.** O onboarding de servidor é sempre interativo.

**Correção:**
```bash
sudo systemctl stop paperclip
sudo rm /home/paperclip/.paperclip/instances/default/config.json
sudo -u paperclip bash -c '
  cd /home/paperclip
  set -a; source /home/paperclip/paperclip.env; set +a
  npx paperclipai onboard   # SEM --yes
'
# No wizard: authenticated / public / https://paperclip.example.com
sudo systemctl start paperclip
```

### 5.6 `sudo -u paperclip` falhava por diretório sem permissão

**Erro:**
```
npm error code EACCES
npm error syscall spawn sh
npm error path /home/ubuntu
```

**Causa:** o diretório de trabalho era `/home/ubuntu`, inacessível para o usuário `paperclip`.

**Correção:** sempre incluir `cd /home/paperclip` no início do comando sudo:
```bash
# ERRADO
sudo -u paperclip bash -c 'npx paperclipai ...'

# CORRETO
sudo -u paperclip bash -c 'cd /home/paperclip && npx paperclipai ...'
```

### 5.7 Output do onboarding não capturado no startup.sh

O `startup.sh` usa `exec > /var/log/paperclip-startup.log 2>&1`, mas o output do
`sudo -u paperclip bash -c '...'` não apareceu no log. O onboarding rodou sem aparecer no log.

**Decisão:** remover o onboarding do `startup.sh`. O onboarding deve ser feito manualmente
via SSH após o boot. O script apenas instala os pré-requisitos e cria o serviço systemd.

## 6. Sequência de onboarding manual (confirmada em produção)

```bash
# 1. Para o serviço (se estiver em restart loop)
sudo systemctl stop paperclip

# 2. Remove config antiga se existir
sudo rm -f /home/paperclip/.paperclip/instances/default/config.json

# 3. Onboarding interativo
sudo -u paperclip bash -c '
  cd /home/paperclip
  set -a; source /home/paperclip/paperclip.env; set +a
  npx paperclipai onboard
'
# Responder no wizard:
#   Deployment mode: authenticated
#   Exposure: public
#   Base URL: https://paperclip.example.com

# 4. Sobe o serviço
sudo systemctl start paperclip
sudo journalctl -u paperclip -f

# 5. Cria o primeiro usuário CEO
sudo -u paperclip bash -c '
  cd /home/paperclip
  set -a; source /home/paperclip/paperclip.env; set +a
  npx paperclipai auth bootstrap-ceo
'
```
